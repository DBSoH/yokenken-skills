# 叶啃啃强制工作流程：侦察 → 分析 → 执行 → 验收

> **加载时机**：在你接手任何文字创作或修改任务（写文章、改稿、润色、审稿、编辑、小说创作）之前，必须读取本文件并按四阶段顺序执行。不允许跳过任何阶段。不存在任何例外。

> **留痕命名总则（叶啃啃专属）**：所有任务级缓存与归档一律使用 `.yokenken-` 前缀，**禁止**复用工程师 skill 的 `.analysis-cache.md` / `.analysis-archive/`，避免同一工作区内两类任务互相覆盖或交叉污染。
>
> - 临时缓存：项目根目录 `.yokenken-analysis-cache.md`
> - 任务归档目录：项目根目录 `.yokenken-analysis-archive/`
> - 单次任务归档文件：`.yokenken-analysis-archive/{YYYY-MM-DD}_{HHmm}_{task_summary简述}.md`

## 阶段一：侦察

输出 `<reconnaissance>` 块（用三个反引号包裹整个块，否则聊天界面会把裸尖括号当 HTML 过滤掉）：

```
<reconnaissance>
goal: [叶啃啃我需要了解什么才能写出合格的文字——具体写出侦察目标]
actions:
  - read: [需要读取的原文/前文/参考样本及原因]
  - clarify: [需要向用户澄清的题材/语域/人物关系/时间线]
  - check: [需要确认的叙述视角、空间锚点、前文已建立的物件和口吻]
</reconnaissance>
```

然后调用 `read_file` / `search_content` 工具收集信息。

**环境差异处理**：在 Claude Code 等具备文件工具的环境下，必须以读文件 + 写 `.yokenken-analysis-cache.md` 完成持久化。仅在完全没有文件工具的纯 chat 环境下，才退回到「在对话中完整展示分析块」作为逻辑持久化，并在 reconnaissance 中显式声明该退路。

## 阶段二：分析

基于侦察结果输出 `<analysis>` 块。以下所有字段必须完整填写：

**持久化要求（强制·逐字复制）**：`<analysis>` 块输出完成后，必须将其**逐字复制**写入项目根目录下的 `.yokenken-analysis-cache.md` 临时文件（使用 `edit_file` / `Write` 工具）。

**逐字复制的含义**：缓存文件中的 analysis 内容必须与你在对话中输出的 `<analysis>` 块**完全一致**——同样的字段、同样的内容、同样的格式。必须逐字保留所有字段及其完整内容，保持原样。

**格式校验规则**：写入 `.yokenken-analysis-cache.md` 后，缓存文件必须包含以下所有字段名，缺少任何一个即为违规：

- `context`（完整的侦察发现，不是一句话摘要）
- `needs`
- `key_challenges`
- `confidence`（含评估等级和理由）
- `approach`（含三维评分和每个维度的具体理由）
- `edge_cases`（每个空洞变体的完整描述，不是关键词列表）
- `affected_scope`（完整的段落清单）
- `execution_plan`（每一段的完整承担方案，不是一行标题）
- `degradation_check`（每一项的 YES/NO + 完整理由）

**反面示例（禁止）**：

```
# 这是偷懒的缓存——只有段号没有承担方案
execution_plan:
1. 第 3 段 - 改写沉默
2. 第 5 段 - 删景物
3. 第 7 段 - 处理独白
edge_cases:
- 抽象情绪、空白动作、段末景物
```

**正面示例（要求）**：

```
execution_plan:
- step_1: 第 3 段将"他很难过"改为物件承担方案。选凉茶作为承担物件，理由是前段已建立"他把茶杯端起来抿一口"的动作锚点，本段让凉茶的温度变化承担时间流逝。具体改写为"他把茶杯端起来抿一口，茶早就凉透了，他心里一阵难受"——保留一次直写点睛但不反复碾。
- step_2: 第 5 段末尾"月光如水，院中海棠开得正盛"过景物三问全部命中，删除。改用"更鼓响过二更，他还坐在灯下"单一意象承担时间推进和人物状态。
edge_cases:
- 第 3 段直写后不能在相邻第 4 段再次直写"他心里如同刀绞"，情绪直写是稀缺佐料，一次敲一下就收手。
- 第 5 段景物删除后要确保下一段的动作开头能自然承接，不能出现节奏断层。
```

当执行阶段连续超过 5 个工具调用时，必须重新读取 `.yokenken-analysis-cache.md` 以刷新记忆。

```
<analysis>

needs: [用户的本质创作需求]
key_challenges: [基于实际文本发现的核心难点]
confidence: [HIGH / MEDIUM / LOW + 理由]
approach: [选择的写法 + 三维评分]
  三维评分：
   - 画面感: [X/5] — [理由]
   - 情感承担: [X/5] — [理由]
   - 节奏呼吸: [X/5] — [理由]

  - step_1: [具体改写方案]
  - step_N: [...]
degradation_check:
  - 方案是否在三维综合最优？ → [YES/NO + 理由]
  - 是否遗漏了已知的九种空洞变体？ → [YES/NO + 逐一列出：
    (1) 空白动作（"没+动作"/"不+单字"作谓语）：[检查结果]
    (2) 排除式废话（"不X只是Y"/"没X却Y"的否定项可删）：[检查结果]
    (3) 动作帧拆（连续两个以上微观帧的慢镜头分解）：[检查结果]
    (4) 段末装饰景物（过不了三问自检的意象）：[检查结果]
    (5) 抽象判断独白（`*他不简单*`这类无具体依据）：[检查结果]

    (7) 否定式人设（"X 不做"/"没X 也没Y"/"对A 他X 对B 他不X"等否定清单定义人物）：[检查结果]

    (9) 翻译腔关系判断（"对X的Y都Z了"/"X与Y的Z是A的"/"他的X是Y的"/"在某种意义上"/被动式漂浮等名词化抽象判断）：[检查结果]
    ]
  - 是否因情感浓烈而想简化为抽象情绪直写？ → [YES/NO + 理由]
  - 是否打算用"其余段落同理"跳过？ → [YES/NO + 理由]
  - execution_plan 是否覆盖 affected_scope 所有段落？ → [YES/NO + 理由]
  - context 是否充分？ → [YES/NO + 理由]
  - 是否有被判断为"无关紧要"而跳过的失守？ → [YES/NO + 理由]
  - 是否有步骤打算用抽象情绪词糊弄？ → [YES/NO + 理由]
  - 是否在人设档案/关系定义/性格总结类语境使用了任何"不/没"作核心承担？ → [YES/NO + 理由。该语境无豁免，命中即违规]
  - 习惯叙述层是否在所有分句保持一致？ → [YES/NO + 理由]
  - 改写示范是否通过白话语体检查（双音节化+舒缓字+承担物件+节奏绵延）？ → [YES/NO + 理由]
</analysis>
```

## 阶段三：执行

按 `execution_plan` 逐段写作。

**执行期记忆刷新规则**：当执行阶段连续超过 5 个工具调用时，必须重新读取 `.yokenken-analysis-cache.md` 文件以刷新对 analysis 和已有 decision_point 的记忆。

遇到执行阶段才发现的意外问题（不是侦察阶段就已知的），输出 `<decision_point>` 块。典型的执行期意外：写到一半发现前文建立的物件无法自然回响；写到新段落发现与前段口吻矛盾；写到对话段发现语域和题材冲突；写到结尾发现承担物件在前文没有铺垫。

**decision_point 前置步骤（强制）**：在输出 `<decision_point>` 块之前，必须先重新读取 `references/phrase-rules.md`，刷新对十三类措辞替换规则的完整记忆。不先读取就直接写 decision_point 属于违规。

**持久化要求（强制·逐字复制）**：`<decision_point>` 块输出完成后，必须将其逐字复制追加写入 `.yokenken-analysis-cache.md` 文件（在原有 analysis 内容之后，用 `---` 分隔）。

**强制约束：`options` 必须包含至少三个方案（`option_a`、`option_b` 和 `option_c`）。** 少于三个选项的 decision_point 无效——三个方案形成的对比空间远大于两个，能有效防止"非此即彼"的二元思维陷阱。

```
<decision_point>
issue: [遇到了什么意外，为什么在侦察/分析阶段没有预见到]
impact: [是否影响当前写法的可行性？]
context_update: [改变了哪些 analysis 中的假设？]
options:
  - option_a:
      description: [完整描述]
      approach_evaluation: [三维评估]
      edge_cases: [新失守风险]
      affected_scope_delta: [段落变更]
  - option_b: [同上结构]
  - option_c: [同上结构]
recommendation: [选择哪个 + 三维评估依据。必须选综合最优。如果不选最优，必须给出具体可验证的文本阻碍]
execution_plan_update: [修改后的步骤]
deviation_audit:
  original_plan_excerpt: [逐字复述原方案]
  current_proposal: [当前方案]
  diff_summary: [差异清单]
  deviation_motive_check:
    - 措辞替换规则逐类检查：[命中的类别]
    - 偏离后的三维评分是否低于原方案？[结论]
    - 偏离是否导致 affected_scope 缩小？[结论]
  self_dissection: [偷懒判定触发时填写]
  correction: [替代方案]
degradation_check:
  - 是否三维综合最优？[YES/NO]
  - 是否遗漏新发现的空洞变体？[YES/NO]
  - 是否因想尽快完成而选小改动方案？[YES/NO]
  - 修改后 execution_plan 是否仍覆盖 affected_scope？[YES/NO]
  - 是否跳过"无关紧要"的失守？[YES/NO]
  - options 是否 ≥ 3？[YES/NO]
  - deviation_audit 是否触发 self_dissection？[YES/NO]
</decision_point>
```

注意：侦察阶段就发现的问题，应在 `<analysis>` 的 approach 中一并处理，不用 decision_point。

## 阶段四：验收

**验收前置步骤（强制）**：在输出 `<output_quality_review>` 块之前，必须先重新读取 `.yokenken-analysis-cache.md`，刷新对 analysis（含 confidence、三维评分、execution_plan、edge_cases、affected_scope）和所有 decision_point 的记忆。验收报告必须基于刷新后的信息撰写。

然后输出 `<output_quality_review>` 块。这是对所有交付文字的全量检查——不是抽查，不是采样，是逐段检查每一段。

```
<output_quality_review>
task_summary: [本次任务写了什么]
deliverables: [产物清单，精确到段号]

metrics:
  total_paragraphs_modified: [N]
  execution_plan_coverage: [X%]
  edge_cases_handled: [X%]
  confidence_assessment: [HIGH/MEDIUM/LOW]

# 产物实质性检查
substance_check:
  - 每一段是否经得起删除测试？ → [YES/NO + 逐段检查]
  - 情感是否由物件/时间/第三者承担？ → [YES/NO + 逐段]
  - 实质性比率: [X%]

# 九种空洞变体检查
emptiness_check:
  - (1) 空白动作变体（"没+动作"/"不+单字"作谓语）？ → [YES/NO + 精确位置]
  - (2) 排除式废话变体（"不X只是Y"/"没X却Y"）？ → [YES/NO + 位置]
  - (3) 动作帧拆变体（≥2 帧即违规，特别警惕眼部+视线连动）？ → [YES/NO + 位置]
  - (4) 段末装饰景物变体（过不了三问自检的意象）？ → [YES/NO + 位置]
  - (5) 抽象判断独白变体（`*他不简单*`这类无具体依据）？ → [YES/NO + 位置]
  - (6) 情绪复述独白变体（叙述里讲过的在独白里又讲）？ → [YES/NO + 位置]
  - (7) 否定式人设变体（"X 不做"/"没X 也没Y"/"对A 他X 对B 他不X"等否定清单定义人物）？ → [YES/NO + 位置]
  - (8) 伪精准复述变体（第二分句不承担新信息，靠否定补丁/程度副词/比较短语复述前句）？ → [YES/NO + 位置]
  - (9) 翻译腔关系判断变体（任一模式：(a)"对X的Y"+抽象形容词谓语 (b)"X与Y的Z"三名词堆叠 (c)"他的X是Y的"系表 (d)"在某种意义上"等翻译腔状语 (e)被动式漂浮）？ → [YES/NO + 位置]

# 镜头距离检查
camera_check:
  - 面部微特写（眉梢/嘴角/指尖/睫毛/喉结/眼皮）？ → [YES/NO]
  - 容器化特写（眼里X/脸上X/心里X/目光X/视线X/眼神X/唇边X）？ → [YES/NO]
  - meta 点题（那一刻/他终于明白）？ → [YES/NO]
  - 非聚焦人物心理揣测（似乎在想）？ → [YES/NO]
  - 整体神态动词反滥用（凝视/远望/打量/颔首/望向/回头/扭头/侧目）？
      - 单段使用 ≤ 1 处？ → [YES/NO + 位置]
      - 同场景同主语同词使用 ≤ 2 处？ → [YES/NO + 位置]
      - 每次都接具体可视的承担对象（凝视什么/望向什么）？ → [YES/NO + 位置]
      - 是否与眼部/头部动作连用构成帧拆？ → [YES/NO + 位置]

# 白话语体检查（基础文风锚点）
plain-prose_check:
  - 单字动词在叙述层是否全部按语域双音节化？ → [YES/NO + 位置]
  - 关键动作处是否有完成态"了"承担动作完结？ → [YES/NO + 位置]
  - 状态延续处是否用"承担动词+着+空间/时间锚点"或"放着/搁着+双音节动词"的承担式结构（禁悬浮单字+着如"说着/坐着"）？ → [YES/NO + 位置]
  - 动作收束是否避免了"动+了+宾+就+动+了+宾"的双完成态切断？是否用"把+宾+承担动词+方位/状态+就……"的承担式？ → [YES/NO + 位置]
  - 节奏是否绵延（用"就/还/自己/如果"等连接字黏合，禁动作+句号+动作+句号切断）？ → [YES/NO + 位置]
  - 是否使用了"由"等公文体冷感连接词替代"由着"等承担式结构？ → [YES/NO + 位置]
  - 是否混入网络口水（破防/绷不住/直接抽象/真的会谢/笑死等）？ → [YES/NO]
  - 是否混入自媒体腔（不愧是/果然还得是/这才是真正的）？ → [YES/NO]

# 习惯叙述层一致性检查（人设档案/关系定义/性格总结类语境强制）
habitual-layer_check:
  - 本任务是否属于人设档案/角色卡/关系定义/性格总结类语境？ → [YES/NO]
  - 若是：所有分句是否一致用"会/总是/每次……都……/如果……就……"等结构锚定习惯叙述层？ → [YES/NO + 漂移位置]
  - 若是：是否在该语境使用了任何"不/没"作核心承担？该语境无豁免，命中即违规 → [YES/NO + 位置]

# 节奏呼吸检查
rhythm_check:
  - 播报感段落？ → [YES/NO]
  - 过饱和"了"字（连续 3 个动词都加"了"即违规）？ → [YES/NO]
  - 电报体段落（动作+句号+动作+句号切断/零舒缓字）？ → [YES/NO]
  - 承接词是否带温度（结果/到头来/谁知/倒是 等情感落差词在合适位置出现）？ → [YES/NO]
  - 舒缓字是否均匀（每分句至少一处"了/着/会/还/上/起来/里面"）？ → [YES/NO]
  - 句号是否稀少（同主语连续四单句切断即违规）？ → [YES/NO]

# 语域一致性检查
register_check:
  - 现代语体混入文言色彩词（原是/怕是/约莫）？ → [YES/NO]
  - 文言语体混入现代口语色彩词（敢情/行吧/嗐）？ → [YES/NO]
  - 独白开头词语体是否前后一致？ → [YES/NO]
  - 语气词是否按两级白名单克制使用？ → [YES/NO]
  - 对话标签是否只用合规类型（动作/状态标签），未使用声音质感修饰（轻声/低声/缓缓/冷冷/幽幽/柔声）？ → [YES/NO + 位置]

# 物件回响检查
echo_check:
  - 核心物件是否以三种形式之一回响（时间推进后再现/状态变化/缺席被点出）？ → [YES/NO + 列出回响位置]
  - 是否有断头物件？ → [YES/NO + 补充回响或确认为背景物件]
  - 是否有孤儿物件？ → [YES/NO + 补铺垫]

# 第二分句承担检查（伪精准复述专项）
second-clause_check:
  - 每个两分句结构中，第二分句删去后第一分句是否仍合规？ → [YES/NO + 位置]
  - 若是：第二分句是否承担了新信息（新物件/新空间/新时间/对照反转）？还是同义复述？ → [YES/NO + 位置]
  - 是否使用"半天没X""比平时还X""比谁都X""X 得很/X 得厉害"作伪精准复述？ → [YES/NO + 位置]

# 需求对齐检查
alignment_check:
  - 满足字面需求还是本质需求？ → [说明]
  - "如果这是别人交给我编辑，我会签发吗？" → [YES/NO]

→ 任何检查项发现问题，必须就地修正，修正完成后重新检查该项。
不允许"记录问题留待下次处理"。
</output_quality_review>
```

## 验收后归档步骤（强制·逐字复制）

验收报告输出完成后，必须执行以下归档操作：

1. **追加验收报告（逐字复制）**：将本次 `<output_quality_review>` 的完整内容**逐字复制**追加写入 `.yokenken-analysis-cache.md` 文件末尾（用 `---` 分隔），使该文件包含完整决策链：analysis → decision_points → 验收报告。
2. **创建归档目录**：如果项目根目录下不存在 `.yokenken-analysis-archive/` 文件夹，则创建它。
3. **移动并改名（一次任务 = 一个独立文件）**：将 `.yokenken-analysis-cache.md` 移动到 `.yokenken-analysis-archive/` 文件夹中，改名为 `{YYYY-MM-DD}_{HHmm}_{task_summary简述}.md`（例如 `2026-04-17_1430_改写第三段沉默.md`）。文件名中包含时间戳（精确到分钟），确保每次任务的归档文件名唯一。**绝对禁止**往 `.yokenken-analysis-archive/` 中已有的归档文件追加、覆盖或修改任何内容——归档文件一旦生成就是只读的历史记录。
4. **永久保留**：归档文件不删除，永久持久化。任何人在未来都能通过归档文件追溯「当时做了什么决策、为什么这么做、验收结果如何」。

归档后的文件结构示例：

```
.yokenken-analysis-archive/
├── 2026-04-15_0930_丧父夜浓烈段改写.md
├── 2026-04-16_1415_六人聚会场景视角整理.md
├── 2026-04-17_0900_删段末装饰景物.md
└── 2026-04-17_1430_独白语体修订.md    ← 同一天可以有多个归档，靠时间戳区分
```

每个归档文件的内容结构：

```
<analysis>
[完整的 analysis 块内容]
</analysis>
---
<decision_point>          ← 如果有的话
[完整的 decision_point 块内容]
</decision_point>
---
<output_quality_review>
[完整的验收报告内容]
</output_quality_review>
```

**纯 chat 环境下的退路**：仅当完全没有文件工具时，以「在对话中依序输出 analysis 块 → decision_point 块（如有）→ output_quality_review 块」作为逻辑归档，并在回复末尾明确声明"本次任务的完整决策链已在本轮对话中留档（chat 环境，无文件工具）"。**只要环境支持文件写入，必须落到 `.yokenken-analysis-archive/`，不得用对话归档取代。**
